# 3D-AI Lab 仕様書

## 0. 位置づけ

本仕様書は、大学1年生向けの「AI駆動開発への導入」を目的とした **3D-AI Lab** の初期仕様である。

3D-AI Lab は、単なるAIチューターではない。  
目的は、学生に「先生役AIから教わる」体験を提供することではなく、**面白いツールを触っているうちに、学生自身が『Codexを使えば自分でもこういう道具を作れるのでは』と気づくこと**である。

したがって、本プロジェクトの本質は次の通りである。

- 学生が面白いツールを使って遊ぶ。
- 条件を変えたり、シミュレーションを観察したりする。
- そのツールが Codex によって作れる、あるいは改造できることを知る。
- 自分の欲しいツール・学習支援ツール・実験環境を Codex と一緒に作り始める。

このツールは、学生を「AIツールの消費者」から「AIと一緒に道具を作る人」へ移行させるための入口である。

---

## 1. 目標

### 1.1 教育上の目標

3D-AI Lab の教育上の目標は以下である。

1. Python基礎を学ぶ学生に、変数・条件分岐・繰り返し・関数・リスト・ログ記録などが、動くシミュレーション世界の中でどう使われるかを体感させる。
2. 学生が「自分のアイディアを動くものにできる」という感覚を持つ。
3. 後期のAI駆動開発・Vibe Coding・Codex利用への導線を作る。
4. GitHubフローや要件定義をいきなり教えるのではなく、先に「作りたい」という動機を作る。
5. 学生が最終的に、自分用の小さなツールを Codex と一緒に作れるようになる。

### 1.2 ツールとしての目標

3D-AI Lab は、ブラウザ上で動く3D実験場として実装する。

学生は画面上のチャット欄または将来的には音声入力を通じて、3D空間内のラボ助手に対して次のような依頼を行う。

- 「ボールが重力で落ちるシミュレーションを作って」
- 「障害物を避けるエージェントを表示して」
- 「エージェントの数を100体に増やして」
- 「迷路探索の動きを見たい」
- 「報酬を変えたらどうなるか比較したい」
- 「このシミュレーションを少し改造したい」

ラボ助手は、依頼内容を実験仕様に分解し、必要に応じて Codex が実装・修正・実行できる形にする。

## 2. 想定ユーザー

### 2.1 主対象

- 大学1年生
- 前期でPython基礎を学習中、または学習済み
- プログラミング初心者
- AI駆動開発やGitHubフローにはまだ慣れていない
- 「自分でアプリやツールを作れる」という感覚はまだ薄い

### 2.2 教員側の利用目的

教員はこのツールを以下の用途で使う。

- Python基礎の動機づけ
- AI駆動開発の前段階としての体験教材
- 学生にCodex活用の可能性を見せるデモ
- 後期授業での「遊ぶ → 改造する → 作る → 自分の道具にする」導線の起点
- 将来的なAI駆動シミュレーション実験教材の基盤

## 3. 基本コンセプト

### 3.1 キーワード

- AI駆動シミュレーション・ラボ
- Prompt-to-Simulation
- AI付き実験場
- Codex-backed Simulation Sandbox
- 遊ぶ → 改造する → 作る → 自分の道具にする
- Pythonを、AIと一緒に作るための共通語として位置づける

### 3.2 「AIチューター」ではなく「ラボ助手」

画面内の3Dキャラクターは、厳密には先生役ではない。

役割は「教える人」ではなく、次のようなラボ助手である。

- 学生のやりたいことを聞く
- 実験可能な仕様に分解する
- 変更案を提案する
- 必要に応じてCodexに実装タスクを渡す
- 実行結果を説明する
- 次に試す実験を提案する

望ましい口調は、講義調ではなく、実験に付き合う助手のような口調である。

例：

> その実験なら、まずエージェント・障害物・ゴールを置くとよさそうです。  
> 今回は行動を「前進・左回転・右回転」にして、壁に当たったらマイナス報酬にしてみましょう。

## 4. 授業内での位置づけ

### 4.1 1年前期

前期では、3D-AI Lab を「作らせる対象」ではなく「触らせる環境」として扱う。

目的はPython基礎の動機づけである。

対応例：

| Pythonの学習内容 | 3D-AI Labでの体験例 |
|---|---|
| 変数 | 速度、重力、個体数、摩擦、報酬値を変える |
| if文 | 壁に当たったら跳ね返る、ゴールしたら停止する |
| for文 | 複数の粒子・エージェントを動かす |
| 関数 | `move_agent()` や `check_collision()` の役割を見る |
| リスト | 複数の障害物やエージェントを管理する |
| CSV/グラフ | 実験ログを保存して可視化する |

前期の評価は軽くする。

例：

1. 何を動かしたか
2. どの値を変えたか
3. 結果はどう変わったか
4. Pythonで学んだどの考え方と関係していそうか

### 4.2 1年後期

後期では、3D-AI Lab をAI駆動開発への入口として使う。

流れ：

1. 3D-AI Labで遊ぶ
2. 既存シミュレーションを少し改造する
3. Codexで小さな変更を加える
4. 自分用の小ツールを作る
5. Git/GitHubで保存・共有・改善する
6. 自由制作・発表を行う

重要なのは、GitHubフローや設計論を最初に主役にしないことである。  
先に「自分の道具が作れる」という体験を作り、その後でバージョン管理や共有の必要性を導入する。

## 5. MVPの技術方針

### 5.1 初期実装方針

初期MVPは以下の構成を推奨する。

- フロントエンド：HTML / JavaScript / Three.js
- バックエンド：Python / FastAPI
- LLM接続：バックエンド経由
- Codex連携：当初は手動または半自動
- シミュレーション結果：JSONでフロントエンドに渡す
- 実行環境：ローカルPCまたは学内サーバ

理由：

- ブラウザで動くため授業配布しやすい
- Three.js は3D表示に十分
- CodexがHTML/JS/Python/FastAPI構成を扱いやすい
- APIキーをフロントエンドに置かずに済む
- GodotやUnityよりMVPが軽量
- Blenderは3D素材制作には有効だが、初期MVPの実行環境としては重い

### 5.2 将来候補

| 候補 | 用途 |
|---|---|
| Three.js / Babylon.js | WebベースMVP、本命候補 |
| Godot | 3D空間・GUI・物理をより本格化する場合 |
| Unity | Mind Render / AI Drill 系教材との連携を重視する場合 |
| Blender | アバターや3D素材制作 |
| Mind Render | 既存教材・比較対象・外部連携候補 |

## 6. システム構成

### 6.1 全体構成

```text
学生
  ↓
ブラウザ上の3D-AI Lab
  - 3D空間
  - ラボ助手アバター
  - チャット欄
  - 吹き出し表示
  - シミュレーション表示
  ↓ HTTP / WebSocket
Python FastAPIバックエンド
  - ユーザー発話の受信
  - LLMへの問い合わせ
  - Codex向けタスク生成
  - シミュレーション実行
  - ログ保存
  ↓
Codex管理領域
  - シミュレーションコード
  - パラメータ
  - 実験ログ
  - 描画用JSON
  ↓
フロントエンド
  - 結果を3D描画
```

### 6.2 重要な設計原則

Codexに3Dエンジン本体を毎回改造させない。

3D表示部分は安定した「殻」として固定する。  
Codexが主に扱うのは以下に限定する。

- シミュレーションロジック
- パラメータ
- 報酬関数
- 初期条件
- 実験ログ生成
- 可視化用JSON生成

これにより、Codexによる変更の事故を抑える。

## 8. 初期MVPの範囲

### 8.1 MVPで必ず実装する機能

1. ブラウザ上に3D空間を表示する
2. 3D空間内にラボ助手アバターを表示する
3. 画面内にチャット入力欄を表示する
4. 学生の入力に対して、ラボ助手が吹き出しで返答する
5. バックエンドに問い合わせて返答を生成する
6. 少なくとも1つのシンプルなシミュレーションを3D表示する
7. パラメータ変更をチャットまたはUIで受け付ける
8. シミュレーション結果をJSONで保存する
9. 結果JSONを読み込んで3Dに反映する

### 8.2 MVPの最初のシミュレーション題材

最初は強化学習ではなく、軽量なルールベースのシミュレーションから始める。

候補：

1. 重力で落ちるボール
2. 壁に当たると跳ね返る粒子
3. 複数エージェントのランダムウォーク
4. 迷路探索
5. 群れシミュレーション
6. 簡単な障害物回避

推奨初期題材：

- `gravity_ball`
- `maze_agent`
- `flocking`

強化学習は、後続フェーズで追加する。

## 9. シミュレーション仕様

### 9.1 シミュレーションはJSON出力を持つ

各シミュレーションは、フロントエンド描画用の `result.json` を生成する。

例：

```json
{
  "meta": {
    "simulation_name": "gravity_ball",
    "steps": 300,
    "dt": 0.016
  },
  "objects": [
    {
      "id": "ball_1",
      "type": "sphere",
      "radius": 0.3,
      "color": "blue"
    }
  ],
  "frames": [
    {
      "t": 0,
      "objects": {
        "ball_1": {
          "position": [0, 5, 0]
        }
      }
    },
    {
      "t": 1,
      "objects": {
        "ball_1": {
          "position": [0, 4.99, 0]
        }
      }
    }
  ]
}
```

### 9.2 フロントエンドは汎用ビューアにする

フロントエンドは、個別シミュレーションごとの特殊実装をできるだけ避ける。

基本的には、`result.json` を読み、以下を描画する。

- 球
- 箱
- 平面
- 線
- 軌跡
- エージェント
- ゴール
- 障害物

### 9.3 シミュレーションコードの要件

各シミュレーションは、以下を満たす。

- `config.json` でパラメータを設定できる
- 実行すると `result.json` を生成する
- 外部ライブラリへの依存を最小限にする
- Codexが改造しやすいように、短く読みやすく書く
- 学生が読めるようにコメントを付ける

## 10. ラボ助手の仕様

### 10.1 役割

ラボ助手は、画面内に表示される3Dアバターである。  
学生とチャットで対話し、必要に応じて吹き出しで返答する。

### 10.2 ラボ助手の振る舞い

ラボ助手は次の原則で返答する。

- 学生のアイディアを否定しない
- すぐに高度な理論説明をしない
- まず実験可能な形に分解する
- 変更可能なパラメータを示す
- 「次に1つだけ変えるなら何か」を提案する
- コードを丸投げするより、実験の観点を示す
- 学生が希望すれば、Codex向けの実装依頼文に変換する

### 10.3 ラボ助手のシステムプロンプト案

```text
あなたは3D-AI Labのラボ助手です。
あなたの役割は、学生に一方的に教える先生ではなく、学生の「作りたい」「試したい」を実験可能な形にすることです。

学生の発話を受け取ったら、次の観点で整理してください。

1. 何を3D空間に置くか
2. 何がどう動くか
3. どのパラメータを変えられるか
4. 何を観察すればよいか
5. 最初に作る最小版は何か
6. 必要ならCodexに渡す実装タスクは何か

返答は初心者向けにしてください。
専門用語を使う場合は、必ず短く説明してください。
最初から大規模な実装を提案せず、まず小さな実験に分解してください。
```

## 11. Codex連携仕様

### 11.1 Codexの役割

Codexは、学生と直接会話する主役ではなく、裏側の実装担当である。

Codexが担当すること：

- シミュレーションコードを書く
- 既存シミュレーションを改造する
- パラメータを追加する
- `config.json` を更新する
- `result.json` の形式に合う出力を生成する
- 実験ログを保存する
- READMEや説明文を更新する
- テストを追加する

### 11.2 Codexに直接触らせない方がよいもの

初期段階では、Codexに以下を頻繁に改造させない。

- 3Dビューア本体
- 認証・APIキー管理
- フロントエンドの基盤設計
- バックエンドのセキュリティ周り
- 大規模な依存関係
- 既に安定している共通UI

### 11.3 Codexへのタスク依頼テンプレート

```text
あなたは3D-AI Labの実装担当です。

目的:
{目的}

現在の対象シミュレーション:
{simulation_name}

変更したい内容:
{requested_change}

守るべき制約:
- 既存のresult.json形式を壊さない
- config.jsonで変更可能なパラメータを増やす
- フロントエンドの共通ビューアは極力変更しない
- 初心者が読めるようにコメントを付ける
- 実行後にresult.jsonが生成されることを確認する

作業内容:
1. 変更対象ファイルを特定する
2. 最小限の変更案を示す
3. コードを変更する
4. 実行してresult.jsonを生成する
5. 変更内容をREADMEに追記する
6. 可能なら簡単なテストを追加する
```

## 12. 権限レベル

授業利用時は、AIの権限を段階的に分ける。

### レベル1：説明のみ

ラボ助手は質問に答えるだけ。  
コードは変更しない。

### レベル2：提案まで

ラボ助手は変更案やCodex向けタスク文を作る。  
ただし実際のコード変更は行わない。

### レベル3：実装まで

Codexが実際にコードを変更し、シミュレーションを実行する。  
授業では原則として、学生または教員の承認後に反映する。

## 13. 強化学習対応は第2段階以降

### 13.1 強化学習を最初から入れない理由

強化学習は魅力的だが、1年生前期の並行教材としては重い。

以下の概念が絡む。

- 状態
- 行動
- 報酬
- 方策
- 探索
- 学習率
- 割引率
- エピソード
- 評価指標

したがって、MVPではルールベースの3Dシミュレーションから始める。  
強化学習は後期または発展題材として追加する。

### 13.2 強化学習シミュレーションの例

`rl_car` の例：

- 状態：車の位置、向き、ゴールまでの距離、壁との距離
- 行動：前進、左回転、右回転
- 報酬：ゴールに近づくとプラス、壁にぶつかるとマイナス
- 終了条件：ゴール到達、衝突、時間切れ
- 評価：成功率、平均報酬、到達時間

ラボ助手は、学生の自然言語のアイディアをこの形式に変換できるようにする。

## 14. UI仕様

### 14.1 画面構成

初期MVPの画面は以下とする。

- 左側または中央：3Dビューポート
- 右側：チャット欄
- 下部：シミュレーション操作パネル
- 3D空間内：ラボ助手アバター
- アバター上部：吹き出し

### 14.2 操作パネル

最低限の操作：

- シミュレーション選択
- 再生
- 一時停止
- リセット
- ステップ送り
- 速度変更
- パラメータ変更
- 実験ログ保存

### 14.3 チャット欄

チャット欄でできること：

- 質問する
- 実験案を相談する
- パラメータ変更を依頼する
- 結果の説明を求める
- Codex向けタスク文を作らせる

### 14.4 音声対応

音声対応は初期MVPでは必須ではない。  
将来的に以下を追加する。

- 音声入力
- 音声読み上げ
- アバターの口パク
- 感情表現
- 表情変化

## 15. ログ仕様

### 15.1 保存するログ

以下を保存する。

- 学生の発話
- ラボ助手の返答
- 実行したシミュレーション名
- 変更したパラメータ
- 実行時刻
- 生成された `result.json`
- Codexに渡したタスク文
- Codexによる変更概要
- 実験結果の概要

### 15.2 学習レポートへの利用

ログは、学生の学習レポートに利用できる。

レポート項目：

1. 何を作ろうとしたか
2. どの条件を変えたか
3. 変えた理由は何か
4. 結果はどう変わったか
5. なぜそうなったと考えるか
6. 次に何を試したいか

## 16. セキュリティ・運用上の注意

### 16.1 APIキー

- APIキーをフロントエンドに埋め込まない
- `.env` に保存する
- `.env.example` のみリポジトリに含める
- 学生PCで運用する場合は、教員管理の中継サーバを使うことを検討する

### 16.2 Codexの実行範囲

Codexが任意のコードを無制限に実行しないようにする。

最低限の方針：

- 作業ディレクトリを限定する
- 実行可能なコマンドを限定する
- 外部通信を必要最小限にする
- ファイル削除やシステム変更を制限する
- 学生利用時は承認制にする

### 16.3 授業利用時の注意

- 学生に「AIが作るからPythonは不要」と誤解させない
- Pythonは、AIと一緒に作るための共通語であると説明する
- コードの丸投げではなく、観察・仮説・比較・説明を評価する
- 初期段階では自動実装よりも提案・改造支援を中心にする

## 17. 開発ロードマップ

### Phase 0：設計確認

- 本仕様書を読む
- MVPをWeb + Three.js + FastAPIで作る方針を確認する
- 既存のMind Render連携は必須にしない
- Blenderは素材制作用途とする

### Phase 1：静的3Dラボ

- ブラウザで3D空間を表示
- 床、ライト、カメラを配置
- 簡単なアバターを配置
- 吹き出し表示を実装
- チャット欄を作る
- 固定応答を返す

完了条件：

- 学生がチャット欄に入力すると、アバターが吹き出しで返答する

### Phase 2：バックエンド連携

- FastAPIサーバを作る
- `/chat` エンドポイントを作る
- フロントエンドから問い合わせる
- LLMなしの仮応答を返す
- ログを保存する

完了条件：

- フロントエンドとバックエンドが通信できる
- 会話ログが保存される

### Phase 3：LLMラボ助手

- LLMクライアントを追加
- ラボ助手システムプロンプトを使う
- 学生の発話を実験仕様に分解する
- Codex向けタスク文を生成できるようにする

完了条件：

- 学生の自然言語入力に対し、ラボ助手が実験案を返す

### Phase 4：シミュレーション表示

- `gravity_ball` を実装
- `result.json` を生成
- フロントエンドで読み込む
- 3D空間内で再生する

完了条件：

- JSONに基づいてボールの動きが3D表示される

### Phase 5：パラメータ変更

- `config.json` を編集できるようにする
- チャットまたはUIからパラメータ変更を受け付ける
- シミュレーションを再実行する
- 結果を再描画する

完了条件：

- 「重力を強くして」などの依頼で結果が変わる

### Phase 6：Codex連携

- ラボ助手がCodex向けタスクを生成する
- Codexがシミュレーションコードを変更する
- 実行して `result.json` を更新する
- READMEを更新する

完了条件：

- Codexが既存シミュレーションに小さな変更を加えられる

### Phase 7：発展シミュレーション

- `maze_agent`
- `flocking`
- `infection_sim`
- `rl_car`

などを追加する。

## 18. 受け入れ条件

初期MVPの受け入れ条件は以下である。

1. ブラウザで起動できる
2. 3D空間が表示される
3. アバターが表示される
4. チャット欄に入力できる
5. アバターが吹き出しで返答する
6. バックエンドと通信できる
7. 少なくとも1つのシミュレーションを表示できる
8. シミュレーション結果がJSONで保存される
9. `config.json` の値を変えると結果が変わる
10. READMEに起動方法が書かれている
11. APIキーをフロントエンドに置いていない
12. Codexが改造しやすいディレクトリ構成になっている

## 19. 最初にCodexへ依頼するタスク

最初にCodexへ渡す依頼文は以下とする。

```text
このspec.mdに基づいて、3D-AI LabのPhase 1〜Phase 2までを実装してください。

要件:
- フロントエンドはHTML/JavaScript/Three.js
- バックエンドはPython/FastAPI
- ブラウザ上に3D空間を表示する
- 簡単なラボ助手アバターを表示する
- チャット欄を表示する
- 学生が入力した文章をFastAPIの/chatへ送信する
- バックエンドは仮応答を返す
- アバター上部に吹き出しで返答を表示する
- 会話ログをlogs/sessions/に保存する
- README.mdに起動手順を書く
- APIキーはまだ使わない
- 将来LLM連携しやすいようにllm_client.pyを空実装で用意する

まずは小さく動くものを優先してください。
```

## 20. 将来像

最終的には、学生が次のような体験を得ることを目指す。

1. 面白い3Dツールで遊ぶ
2. 条件を変える
3. 動きの変化を観察する
4. 「これを自分でも作れるのでは」と気づく
5. Codexに相談して改造する
6. 自分用の小さなツールを作る
7. そのツールを使ってさらに学ぶ

この流れにより、学生はプログラミングを「課題を解くための作業」ではなく、**自分の環境を変えるための手段**として捉え始める。

3D-AI Lab は、そのための入口である。
